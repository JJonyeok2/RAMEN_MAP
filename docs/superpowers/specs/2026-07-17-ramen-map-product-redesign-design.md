# RAMEN MAP 제품·애플리케이션 재설계

- 작성일: 2026-07-17
- 상태: 구현 전 사용자 검토
- 대상 릴리스: 지도 API 없이 동작하는 실데이터 기반 V1

## 1. 제품 목표

RAMEN MAP은 사용자가 현재 상황에 맞는 라멘집과 실제 판매 메뉴를 결정하도록 돕는 탐색·추천 서비스다. 핵심 성공 기준은 사용자가 빠른 찾기 흐름에 진입한 뒤 60초 안에 매장을 선택하거나 길찾기를 누르는 것이다.

제품은 서로 다른 두 사용 의도를 분리한다.

1. **배고파요 · 빨리 찾기**: 현재 위치에서 지금 갈 만한 라멘집을 빠르게 결정한다.
2. **라멘 탐방**: 지역과 메뉴 취향을 기준으로 새로운 라멘집과 한 그릇을 천천히 탐색한다.

운영 서비스에는 실제 매장 데이터만 노출한다. 창작 데모 데이터는 자동화 테스트의 fixture로만 남긴다.

## 2. 확정된 제품 결정

- 모바일 우선으로 설계하고 데스크톱에서는 동일한 흐름을 넓은 레이아웃으로 제공한다.
- 위치 반경은 3km에서 시작하고 후보가 부족하면 10km, 최대 30km까지 자동 확장한다.
- 사용자는 `취향 우선`, `균형 추천`, `가까운 곳 우선`을 직접 선택할 수 있다.
- 원하는 한 그릇은 빠른 선택 버튼과 자연어 대화를 결합해 표현한다.
- 검증 대기 매장도 공개하지만 검증 완료 매장과 시각적·논리적으로 구분한다.
- 챗봇은 검증 완료 매장을 먼저 추천하고 부족할 때 검증 대기 후보를 별도 영역으로 보완한다.
- V1은 로그인 없이 모든 소비자 기능을 사용할 수 있다.
- 선택 로그인, 찜, 개인 취향, 방문 기록은 후속 릴리스로 분리한다.
- 카카오맵은 심사 완료 후 연결한다. V1은 지도 SDK 없이도 완전하게 사용할 수 있어야 한다.

## 3. V1 범위

### 포함

- 실데이터 전용 공개 화면
- 빠른 찾기와 라멘 탐방의 분리된 진입 흐름
- 현재 위치 또는 저장된 지역 중심점 기준 반경 검색
- 메뉴 단위 취향 필터와 결정론적 추천 점수
- 자연어 취향 해석과 추천 이유 설명
- 매장 목록, 메뉴 목록, 상세 화면, 출처와 확인일
- 검증 완료·대기·재확인 필요·폐업/이전 상태
- 관리자용 수집·검증·수정 화면
- 현재 8개 후보 데이터의 정규화 마이그레이션

### 후속 릴리스

- 카카오맵 또는 다른 지도 공급자 연결
- 실제 도로 이동시간과 경로 계산
- 선택 로그인, 찜, 취향 기억, 방문 기록
- 일반 사용자 제보와 커뮤니티 검증
- 전국 자동 수집·대량 등록

## 4. 정보 구조

### 공개 영역

- `/`: 두 탐색 모드를 고르는 홈
- `/nearby`: 배고파요 · 빨리 찾기
- `/explore`: 라멘 탐방
- `/shops/[slug]`: 매장과 메뉴 상세

### 운영 영역

- `/admin`: 검증 현황 대시보드
- `/admin/branches`: 후보 지점 수집·검증
- `/admin/branches/[id]`: 지점, 메뉴, 출처, 이력 편집

일반 사용자 내비게이션에서 관리자 링크를 제거한다. 공개 배포 전 관리자 페이지와 쓰기 API에는 서버 측 운영자 인증을 반드시 적용한다.

## 5. 사용자 흐름

### 5.1 홈

홈은 복잡한 지도와 필터를 보여주지 않는다. 두 개의 주요 행동만 강조한다.

- `배고파요 · 빨리 찾기`: 지금 갈 수 있는 가까운 라멘집
- `라멘 탐방`: 지역과 취향으로 새로운 한 그릇 탐색

### 5.2 빠른 찾기

1. 브라우저에 현재 위치 사용을 요청한다.
2. 거절·실패 시 동네나 역을 선택한다. V1의 수동 위치는 `areas` 테이블에 저장한 지역 중심 좌표를 사용한다.
3. 3km 안의 공개 가능한 실제 지점을 조회한다.
4. 결과가 3곳 미만이면 10km, 다시 부족하면 30km로 확장한다.
5. 기본 추천 모드는 `가까운 곳 우선`이다.
6. 영업 상태, 직선거리, 추천 메뉴, 검증 상태를 담은 상위 3곳을 제공한다.
7. 사용자는 `다시 골라줘`, `취향 추가`, `길찾기` 중 하나를 선택한다.

V1은 경로 API가 없으므로 이동시간을 추정하지 않는다. 모든 거리는 `직선거리`로 명시한다.

### 5.3 라멘 탐방

1. 지역 또는 수동 위치를 선택한다.
2. 메뉴 유형, 국물 스타일, 농도, 맵기, 육수 베이스를 빠른 선택으로 설정한다.
3. 자연어로 추가 조건을 입력할 수 있다.
4. 세 가지 추천 모드 중 하나를 고른다.
5. 결과 목록에서 메뉴 단위 추천 이유를 비교한다.
6. 매장 상세에서 출처와 마지막 확인일을 확인하고 길찾기 또는 공유를 선택한다.

## 6. 도메인 데이터 모델

매장 한 행에 대표 메뉴를 함께 저장하는 기존 구조를 폐기하고, 지점과 메뉴를 독립된 엔터티로 관리한다.

### `shops`

브랜드 또는 매장의 정체성이다.

- `id`, `slug`, `name`, `normalized_name`
- `created_at`, `updated_at`

### `branches`

사용자가 실제로 방문하는 지점이다.

- `id`, `shop_id`, `branch_name`
- `region`, `district`, `address`, `lat`, `lng`
- `phone`, `public_status`, `verification_status`
- `last_verified_at`, `created_at`, `updated_at`

### `menu_items`

추천 계산의 기본 단위다.

- `id`, `branch_id`, `name`, `price`
- `availability_status`, `verification_status`
- `last_verified_at`, `created_at`, `updated_at`

### `menu_profiles`

메뉴의 취향 속성이다.

- `menu_item_id`
- `ramen_types`: 쇼유, 시오, 미소, 돈코츠, 츠케멘, 마제소바
- `broth_style`: 청탕, 백탕, 비빔, 츠케
- `body_level`: 1~5
- `spiciness_level`: 0~5
- `broth_bases`: 닭, 돼지, 소, 해산물, 채소
- `tags`

### `opening_hours`와 `opening_exceptions`

요일별 영업시간, 브레이크, 라스트오더와 임시 휴무를 구조화한다. 정보가 없으면 `unknown`으로 유지하며 영업 중이라고 추정하지 않는다.

### `source_evidence`

각 사실을 뒷받침하는 근거다.

- 대상 엔터티와 필드
- 출처 이름, URL, 확인일
- 근거 메모와 수집자

### `verification_events`

승인, 보류, 충돌, 폐업, 이전과 필드 변경 이력을 기록한다. 현재 값을 덮어쓰기 전에 이전 값을 이력에 남긴다.

### `areas`

지도·외부 지오코더 없이 수동 위치를 선택하기 위한 지역과 역 중심 좌표를 저장한다. V1은 안양, 망원, 홍대, 합정과 해당 생활권의 주요 역 중심점을 포함한다.

## 7. 데이터 신뢰 정책

### 검증 상태

- `verified`: 근거와 확인일이 있고 운영자가 승인
- `candidate`: 출처가 있으나 최종 확인 전
- `stale`: 검증 후 시간이 지났거나 근거가 충돌
- `rejected`: 라멘 전문점이 아니거나 중복·오류 후보

### 공개 상태

- `active`: 일반 검색과 추천 대상
- `hidden`: 운영자 검토 중이며 공개하지 않음
- `closed`: 폐업이 확인되어 일반 결과에서 제외
- `moved`: 이전한 과거 지점이며 새 지점과 이력으로 연결

주소·영업 정보는 마지막 확인 후 90일, 메뉴·가격은 180일이 지나면 재확인 대상으로 전환한다. 오래된 값은 삭제하지 않고 공개 화면에 마지막 확인일과 상태를 표시한다.

주소나 가격이 충돌하면 최신으로 보이는 값을 자동 채택하지 않는다. 모든 근거를 보존하고 `stale`로 전환한 뒤 운영자가 현재 값을 확정한다.

## 8. 추천 엔진

### 8.1 처리 단계

1. 사용자의 버튼 선택과 자연어를 정규화된 `TasteIntent`로 변환한다.
2. 위치, 반경, 공개 상태로 지점 후보를 만든다.
3. 필수 제외 조건을 적용한다.
4. 각 메뉴와 취향의 일치도를 계산한다.
5. 선택한 추천 모드의 가중치로 메뉴를 점수화한다.
6. 매장별 최고 점수 메뉴를 대표 추천으로 선택한다.
7. 검증 완료 결과와 검증 대기 보조 결과를 분리한다.
8. 점수 근거를 사용자 문장으로 설명한다.

### 8.2 자연어 규칙

- 느끼함 회피 → 낮은 농도와 청탕 선호, 백탕 감점
- 스트레스·화남 → 카라이 또는 높은 맵기 선호
- 매운맛 부정 → 맵기 0~1의 필수 조건
- 찍어 먹기 → 츠케멘 우선
- 비벼 먹기 → 마제소바 우선
- 깔끔한 간장 → 쇼유 청탕 우선

부정과 제외 조건을 긍정 선호보다 먼저 적용한다. 자연어 해석의 확신이 낮으면 임의 추천하지 않고 하나의 짧은 추가 질문을 제시한다.

### 8.3 초기 가중치

| 모드 | 취향 | 거리 | 영업 정보 | 신뢰도 |
|---|---:|---:|---:|---:|
| 취향 우선 | 60 | 20 | 10 | 10 |
| 균형 추천 | 40 | 30 | 20 | 10 |
| 가까운 곳 우선 | 15 | 55 | 20 | 10 |

가중치는 코드 상수가 아닌 설정 객체로 관리하고 테스트 시나리오와 실제 선택 데이터를 통해 조정한다.

### 8.4 검증 상태와 결과 구성

- 검증 완료 결과가 3개 이상이면 상위 3개를 추천한다.
- 검증 완료 결과가 3개 미만이면 부족한 수만큼 후보를 섞지 않고 `검증 전 후보` 섹션으로 따로 제공한다.
- 구조화된 영업시간 계산 결과가 현재 `closed`인 지점은 빠른 찾기에서 제외한다.
- 영업시간을 모르는 지점은 `영업 여부 확인 필요`로 감점하되 완전히 제외하지 않는다.
- 폐업·이전·제외 상태는 모든 일반 검색과 추천에서 제외한다.

### 8.5 결과 부족 처리

1. 3km에서 10km, 이후 30km로 확대한다.
2. 부드러운 선호 조건만 완화 후보로 제안한다.
3. 맵기 제외, 식재료 제외 같은 필수 조건은 자동으로 완화하지 않는다.
4. 완화하거나 후보 데이터를 제시한 사실을 결과 상단에 설명한다.

챗봇은 데이터나 순위를 생성하지 않는다. 사용자의 문장을 `TasteIntent`로 바꾸고 결정론적 엔진의 결과를 설명한다.

## 9. 애플리케이션 구조

UI, 추천 규칙, DB 접근을 분리해 한 파일이 여러 책임을 갖지 않게 한다.

```text
app/
  (public)/
    page.tsx
    nearby/page.tsx
    explore/page.tsx
    shops/[slug]/page.tsx
  admin/
    page.tsx
    branches/page.tsx
    branches/[id]/page.tsx
  api/
    v1/nearby/route.ts
    v1/recommendations/route.ts
    v1/shops/[slug]/route.ts
    admin/...
features/
  location/
  shops/
  recommendation/
  admin/
domain/
  shop.ts
  menu.ts
  taste-intent.ts
  verification.ts
db/
  schema.ts
  repositories/
map/
  map-adapter.ts
  kakao-adapter.ts
```

의존 방향은 `UI/API → 애플리케이션 서비스 → 도메인 → 저장소 인터페이스`로 제한한다. D1과 카카오맵은 외부 구현체이며 추천 도메인이 직접 의존하지 않는다.

## 10. API와 데이터 흐름

### 공개 API

- `GET /api/v1/nearby`: 위치와 반경으로 공개 지점 조회
- `POST /api/v1/recommendations`: 위치, 취향, 추천 모드를 받아 설명 가능한 결과 반환
- `GET /api/v1/shops/[slug]`: 매장, 메뉴, 영업, 출처 상세 반환

### 관리자 API

- 후보 생성과 지점 수정
- 메뉴 추가·수정·판매 중지
- 출처 추가와 충돌 표시
- 상태 승인·보류·재확인·제외

모든 입력은 API 경계에서 검증한다. 공개 API는 DB 행을 그대로 노출하지 않고 공개 DTO로 변환한다. 관리자 쓰기는 인증된 운영자만 허용하고 모든 변경을 `verification_events`에 기록한다.

## 11. 지도 공급자 분리

지도는 탐색 엔진이 아니라 결과를 시각화하는 선택적 UI다.

- V1은 카드 목록과 직선거리로 완전하게 동작한다.
- 위치 검색은 브라우저 좌표 또는 `areas` 중심 좌표를 사용한다.
- 가짜 전국 지도나 실제 지도로 오해할 수 있는 배경은 운영 화면에서 제거한다.
- 카카오 심사가 완료되면 `MapAdapter` 구현만 연결한다.
- 지도 실패 시 목록, 추천, 상세와 외부 지도 검색 링크는 계속 사용할 수 있어야 한다.

## 12. 오류·빈 상태 처리

- 위치 권한 거절: 지역·역 선택으로 전환
- 위치 확인 실패: 마지막 위치를 저장하지 않고 수동 선택 제안
- 결과 없음: 반경 확장 후 완화 가능한 취향 안내
- 영업시간 없음: `영업 여부 확인 필요` 표시
- 데이터 출처 충돌: 확정 문구 대신 충돌 배지와 근거 표시
- DB/API 실패: 기존 데모 데이터로 대체하지 않고 명시적인 재시도 화면 제공
- 자연어 해석 불확실: 한 번에 하나의 짧은 확인 질문 제공

## 13. 개인정보와 보안

- V1은 소비자 로그인을 요구하지 않는다.
- 현재 위치는 추천 요청 처리에만 사용하고 D1에 저장하지 않는다.
- 분석 이벤트에 원본 위도·경도를 기록하지 않고 선택 반경과 지역 코드만 기록한다.
- 관리자 쓰기 API는 공개 배포 전에 서버 측 운영자 인증과 CSRF 방어를 갖춘다.
- 출처 URL과 사용자 입력은 허용된 프로토콜과 길이를 검증한다.

## 14. 테스트 전략

### 도메인 단위 테스트

- 3km → 10km → 30km 확장
- 세 추천 모드의 순위 차이
- 자연어 부정이 긍정 추론보다 우선하는지
- 검증 완료와 후보 결과가 섞이지 않는지
- 폐업·이전·현재 닫힘 처리
- 필수 제외 조건이 자동 완화되지 않는지

### 저장소·마이그레이션 테스트

- 새 정규화 스키마 적용
- 기존 8개 후보의 지점·메뉴·출처 보존
- 변경 이력과 상태 전환
- 중복 매장과 충돌 근거 처리

### API 테스트

- 입력 검증과 공개 DTO
- 위치·반경·취향 조합
- 관리자 인증 없는 쓰기 거부
- DB 장애 시 명시적 오류 반환

### 사용자 흐름 테스트

- 위치 허용 후 60초 이내 상위 3개와 길찾기 도달
- 위치 거절 후 지역 선택으로 동일 흐름 완료
- 라멘 탐방의 버튼·자연어 결합
- 모바일과 데스크톱 핵심 흐름

## 15. 측정 기준

V1의 핵심 지표는 빠른 찾기 진입부터 매장 선택 또는 길찾기 클릭까지의 시간이다. 목표는 완료 사용자의 중앙값 60초 이하다.

보조 지표는 다음과 같다.

- 추천 결과에서 상세 또는 길찾기로 이어지는 비율
- 3km, 10km, 30km별 결과 충족률
- 검증 전 후보가 사용된 비율
- 자연어 추가 질문이 필요한 비율
- 7일 내 재방문율

## 16. 구현 순서

1. 도메인 타입과 추천 인터페이스를 먼저 정의한다.
2. 정규화 D1 스키마와 기존 8개 후보 마이그레이션을 만든다.
3. 저장소 계층과 공개/관리자 DTO를 구현한다.
4. 메뉴 단위 추천 엔진과 자연어 규칙을 테스트 우선으로 구현한다.
5. 빠른 찾기 목록 흐름을 구현한다.
6. 라멘 탐방과 매장 상세를 구현한다.
7. 관리자 검증 화면을 새 스키마에 맞게 교체하고 인증을 적용한다.
8. 데모 데이터를 운영 경로에서 제거하고 테스트 fixture로 이동한다.
9. 전체 테스트와 모바일·데스크톱 렌더링을 검증한 뒤 비공개 배포한다.
10. 카카오 심사 완료 후 별도 설계·계획으로 지도 어댑터를 연결한다.

## 17. V1 완료 조건

- 공개 화면과 API가 창작 데모 매장을 반환하지 않는다.
- 현재 8개 후보가 손실 없이 정규화 DB로 이전된다.
- 빠른 찾기가 위치 또는 지역 중심점에서 3·10·30km를 순차 적용한다.
- 사용자가 세 추천 모드를 선택할 수 있다.
- 버튼과 자연어가 하나의 취향 의도로 결합된다.
- 검증 완료 추천과 검증 전 후보가 분리되어 표시된다.
- 라멘 탐방에서 메뉴 단위 필터·추천·상세 확인이 가능하다.
- 지도 SDK가 없어도 모든 핵심 흐름이 동작한다.
- 관리자 쓰기 경로가 일반 사용자에게 노출되지 않고 서버에서 보호된다.
- 도메인, 마이그레이션, API, 핵심 사용자 흐름 테스트가 모두 통과한다.
